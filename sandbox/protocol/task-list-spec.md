# 任务清单规范（task-list-spec）

`TASKS.md` 是整个循环的**唯一输入**，由人类编写。它是不可变的：循环运行期间主控和 Worker 都**不得修改**此文件（任务状态由 `loop/state.md` 管理，见 [state-spec.md](state-spec.md)）。

## 文件结构

```markdown
# 任务清单：<项目名>

## 全局上下文
<项目背景、技术栈、通用约束。这一节会原样注入到每个任务简报中，
 请控制在 30 行以内，只写所有任务都需要知道的信息。>

## 任务

### [T1] <任务标题>
```yaml
id: T1
depends_on: []
touches:
  - src/data/
acceptance:
  - <可机器/肉眼验证的标准 1>
  - <可验证标准 2>
context_files:
  - docs/requirements.md
constraints:
  - <本任务特有的约束>
```
<正文：用自然语言描述任务目标与背景，写给一个
 "对项目一无所知、只能看到本简报"的执行者。>

### [T2] <下一个任务>
...
```

## 字段定义

| 字段 | 必填 | 说明 |
|---|---|---|
| `id` | 是 | 任务唯一标识，格式 `T<数字>`，与标题中 `[T1]` 一致 |
| `depends_on` | 是 | 依赖的任务 id 列表，空列表 `[]` 表示无依赖 |
| `touches` | 是 | 本任务允许**写入**的文件/目录路径列表（并行安全判定依据） |
| `acceptance` | 是 | 验收标准列表，主控据此验收，Worker 据此自查 |
| `context_files` | 否 | Worker 开工前必须阅读的已有文件 |
| `constraints` | 否 | 任务级约束（禁区、风格要求、性能要求等） |

## 编写规则

1. **验收标准必须可验证**。坏例子："代码质量良好"；好例子："`node --test` 全部通过"、"`index.html` 在浏览器打开无 console 报错"、"文件 `api.md` 包含全部 4 个接口的请求/响应示例"。
2. **`touches` 决定并行**。两个就绪任务的 `touches` 路径互不重叠（且无依赖关系）时，主控可以并行派发它们。写得越精确，并行度越高；不确定就写宽一点（牺牲并行换安全）。
3. **依赖必须无环**。主控启动时会做环检测，发现环会直接升级给用户。
4. **任务粒度**：一个任务 = 一个 Worker 一次上下文内可完成的工作量。经验值：产出 1–5 个文件、或一个内聚的功能模块。太大的任务应在编写清单时就拆开，而不是指望 Worker 自己拆。
5. **正文自包含**：Worker 看不到其他任务的正文，只能看到依赖任务回执中的摘要。所以每个任务正文要独立可读。

## 依赖语义

- `depends_on: [T1, T2]`：T1 和 T2 都 `done` 后本任务才就绪。
- 依赖任务的回执中「关键决策」和「对下游的提醒」两节，会被主控自动注入到本任务的简报中——这是任务间传递信息的**唯一通道**。需要跨任务传递的信息，务必让上游任务写进回执。
