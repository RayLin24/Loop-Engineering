# 状态文件规范与冷启动恢复协议（state-spec）

`loop/state.md` 是循环的**唯一事实源（single source of truth）**。主控的一切决策只依据此文件 + `TASKS.md`，不依赖会话记忆。任何一个新会话读完这两个文件，就能完整接管循环——这就是本框架抗中断、抗上下文膨胀的根基。

主控是**代际制**的：每个会话是一代，最多服务「每代最大轮次」轮即交接退出，下一代读盘接续。因此 state.md 除记录任务状态外，还记录**循环生命周期**（`循环状态` 字段），供外部驱动器（`runner/`）与人类判断"循环现在需要什么"。

## 目录布局（主控首轮初始化）

```
<项目根>/
├── TASKS.md              # 任务清单（人类编写，只读）
├── LOOP.md               # 框架入口提示词
├── runner/               # 可选: 无人值守驱动器（框架自带, 见 runner/README.md）
└── loop/
    ├── state.md          # 状态文件（本规范定义）
    ├── decisions.md      # 全局决策记录（主控验收通过时追加, 见 handoff-spec）
    ├── briefs/           # 主控 → Worker 的任务简报（含 *.verify.md 验证简报）
    │   └── T1.md         #（重试时上次尝试归档为 T1.attempt<n>.md）
    ├── reports/          # Worker → 主控 的回执
    │   └── T1.md
    ├── verdicts/         # Verifier → 主控 的裁决
    │   └── T1.md
    ├── lessons.md        # 经验教训（Meta-Analyst 维护, 只注入 Worker 简报, 见 meta-spec）
    ├── meta/             # 元分析报告（Meta-Analyst → 主控, 按轮次留档）
    │   └── round5.md
    └── FINAL.md          # 终局报告（全部完成后生成）
```

## state.md 格式

```markdown
# Loop State
- 项目: <项目名>
- 清单: TASKS.md (共 N 个任务)
- 更新时间: <ISO 时间>
- 循环状态: running | handoff | awaiting-human | finished
- 轮次: <全循环累计第几轮>
- 代次: <当前是第几代主控会话>
- 本代轮次: <本代已完成几轮, 交接时下一代归零>
- 最大轮次: <保险丝, 默认 3×任务数; 达到时置 awaiting-human 并升级用户; 仅人类可改>
- 上次元分析轮次: <0 表示尚未做过>

## 运行配置（Meta-Analyst 可在 meta-spec 白名单内调整, 主控执行）
- 每批并行上限: 3
- 每任务重试上限: 2
- 每代最大轮次: 3
- lessons 注入 Worker 简报: 开
- 验证抽查强度: 标准

## 任务状态表
| id | 状态 | 重试 | 回执 |
|---|---|---|---|
| T1 | done | 0 | loop/reports/T1.md |
| T2 | dispatched | 0 | - |
| T3 | pending | 0 | - |
| T4 | failed | 1 | loop/reports/T4.md |
| T5 | blocked | 2 | loop/reports/T5.md |

## 循环度量（验收产生结论时更新, Meta-Analyst 的主要输入）
| id | 尝试 | 结局 | 失败类别 |
|---|---|---|---|
| T1 | 1 | done | - |
| T4 | 2 | 重试中 | 验收不达标(测试未过) |
| T5 | 3 | blocked | 越界写入 |

失败类别用固定词表, 便于跨任务统计: `验收不达标` / `越界写入` / `回执缺失` /
`自查诚实❌` / `约束违反` / `环境问题`。一次尝试一行结论, 后续尝试更新同行。
可疑裁决复核推翻的假失败不记入度量（记日志即可）。

## 日志（追加式，每轮一条，只记决策不记过程）
- [轮1] 初始化。派发 T1。
- [轮2] T1 验收通过。并行派发 T2、T3。
- [轮3] T2 通过；T3 验收失败(原因:缺少错误处理)，反思后重试。
- [轮3] 第 1 代交接（本代 3 轮）。
- [轮4] 第 2 代接管，冷启动对账: T3 回执在、裁决缺 → 从验证环节继续。
- [轮5] 元分析: 并行上限 3→1(两次越界均在并行批); lessons 更新。

## 待办备忘（跨轮需要记住的事，可为空）
- T4 的回执提到 API 限流，T6 派发时需在简报中提醒。
- 建议追加任务: T10 的超时收紧影响 X 模块, 需善后（待人类决定）。
```

### 循环状态语义（外部驱动器的判停依据）

| 值 | 含义 | 驱动器动作 |
|---|---|---|
| `running` | 一代主控正在工作（或中途崩溃未及改写） | 等待；若会话已退出则视同 handoff 重启 |
| `handoff` | 本代轮次用满，正常交接 | 立即拉起新会话续跑 |
| `awaiting-human` | blocked 汇总 / 轮次保险丝 / 升级事项，需人工 | 停止，通知人类 |
| `finished` | FINAL.md 已生成 | 停止 |

## 状态机

```
pending ──(依赖满足,派发)──> dispatched ──(验收通过)──> done
                                │
                            (验收失败)
                                ▼
                             failed ──(带反思重试,≤2次)──> dispatched
                                │
                            (重试耗尽)
                                ▼
                             blocked ──(升级用户,人工解决后)──> pending
```

- `pending`：未开始。`dispatched`：简报已写出、Worker 执行中。
- `failed`：验收未通过，等待重试。`blocked`：重试耗尽或外部依赖缺失，需人工介入。
- `done`：验收通过，终态。

## 写入纪律（保证可恢复性的关键）

1. **先写后做**：派发 Worker 之前，先把该任务标记为 `dispatched` 并保存 state.md。这样即使派发过程中断，恢复时也知道"这个任务可能做了一半"。
2. **每轮结束必落盘**：一轮循环的最后一步永远是更新 state.md（状态表 + 日志 + 更新时间 + 轮次 + 本代轮次）。
3. **增量编辑**：状态表更新用单行编辑；日志、度量、备忘只追加；禁止无必要的全文重写——全文重写既把整个文件回显进主控上下文烧预算，又放大写坏文件的风险。
4. **日志只记决策**：一行一条，记录"做了什么调度决定和为什么"，不记执行细节（细节在回执里）。这保证 state.md 永远足够小，可被完整读入。

## 冷启动恢复协议

任何新会话（代际交接、人为中断后重启、上下文耗尽后重启、换工具续跑）按以下步骤接管：

1. 读 `LOOP.md`（了解自己的角色）→ 读 `TASKS.md` → 读 `loop/state.md`。
2. **代际登记**：`代次` +1，`本代轮次` 归零。检查 `循环状态`：
   - `handoff` / `running`（running 意味着上一代中途崩溃）→ 置回 `running`，继续。
   - `awaiting-human` → 先检查触发原因是否已被人工解决（blocked 任务被改回 pending、保险丝被人调大）；未解决则复述升级事项后停止，不得自行绕过。
   - `finished` → 告知用户循环已结束，除非 TASKS.md 有新任务待补入。
3. **对账**：逐一处理非终态任务——
   - `dispatched` 的任务：读它的回执 `loop/reports/<id>.md`。
     - 回执不存在或不完整 → 任务执行中断，重新派发，简报「主控附言」注明「此前有一次被中断的尝试，先检查 touches 下的现有产出再决定重做或续做」。
     - 回执存在且完整，但裁决 `loop/verdicts/<id>.md` 不存在 → 验证环节中断，从"写验证简报、派发 Verifier"一步继续，**不重做任务**。
     - 回执与裁决都存在 → 正常走验收流程的最后一步（依据裁决定 done/failed）。
   - `failed` 的任务：按正常重试流程处理。
4. 在日志中追加一条 `[轮N] 第 <代次> 代接管，冷启动对账: ...`，然后进入正常调度循环。

## 任务清单修订协议（循环中途需求变化时）

`TASKS.md` 对主控和 Worker 只读，但**人类**可以在循环暂停时修订。规则按任务当前状态区分：

- **追加新任务**：任意时刻可以。续跑时主控把新任务补入状态表（`pending`），重新校验依赖无环。主控在备忘中登记的"建议追加任务"，也由人类在此时采纳落笔——主控自己永不写 TASKS.md。
- **修改 `pending` 任务**：允许。续跑时主控重新校验，并在日志记录一条"任务 Tn 已被人工修订"。
- **修改 `done` 任务**：不允许改原任务——已验收的产出基于旧定义。需求变化时追加一个新的修复/改造任务，`depends_on` 指向原任务。
- **修改 `dispatched` / `failed` / `blocked` 任务**：先等其到达稳态（done 或 blocked），或人工把状态改回 `pending` 后再修订。
- 删除任务同理：仅 `pending` 可删（主控续跑时将其从状态表移除，并校验没有其他任务依赖它）。
