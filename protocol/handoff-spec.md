# 主从交接协议（handoff-spec）

主控、Worker 与 Verifier 之间**只通过三种文件**通信：简报（brief，主控→Worker）、回执（report，Worker→主控）、裁决（verdict，Verifier→主控；其输入是主控写的验证简报）。禁止任何隐式信息传递（如"Worker 应该记得上一轮……"——Worker 没有上一轮，每个 Worker、每个 Verifier 都是全新上下文）。

**简报是指针，不是拷贝**。简报只写「阅读清单」——指向信息所在的文件与节；读取与装配由收件的 Worker/Verifier 自己完成。理由是预算不对称：主控是全循环最稀缺的上下文，Worker/Verifier 是一次性上下文——同样的信息装配工作，发生在后者身上近乎免费，发生在主控身上则直接缩短这一代的寿命。主控在简报中亲笔书写的只有判断性内容：调度附言、命令授权、失败反思。

**重试归档规则**：三类文件都按 `<task-id>.md` 命名，指向**当前尝试**。派发重试前，主控必须先把上一次尝试的简报、回执、裁决重命名归档为 `<task-id>.attempt<n>.md`（n 为上次的尝试次数），再写新简报。失败历史因此完整保留——blocked 升级时的失败史汇总、反思的撰写，都以归档文件为依据。

## 简报格式：`loop/briefs/<task-id>.md`

由主控在派发前生成。模板：

```markdown
# 任务简报: <task-id> <任务标题>
- 尝试次数: 1  <!-- 重试时递增 -->

## 阅读清单（按序读取并装配你的认知, 这是你对项目的全部信息来源）
1. TASKS.md：「全局上下文」节 + 「[<task-id>]」任务节（含 yaml 的
   touches/acceptance/constraints/context_files 与正文）。其余任务节是禁区——
   先 Grep 定位行号, 再按行号范围 Read, 不要整读。
2. 上游回执：loop/reports/<依赖id>.md（每个 depends_on 一行;
   只读「关键决策」与「对下游的提醒」两节）  <!-- 无依赖则删除本行 -->
3. loop/decisions.md：全局决策记录（不存在则跳过）
4. loop/lessons.md：经验教训   <!-- 运行配置「lessons 注入」为关时删除本行 -->

## 主控附言  <!-- 调度层提醒; 无则写"无" -->
无

## 上次失败的反思  <!-- 仅重试时包含此节, 主控亲笔 -->
<上次未通过的标准 → 原因分析(以裁决 ❌ 条目与「修复建议」为依据) → 本次应改变的做法>
```

**生成纪律**：

- 简报正文 ≤15 行（「上次失败的反思」不计）。禁止把 TASKS.md、上游回执、decisions.md 的内容抄写进简报——那是主控最常见的预算泄漏。
- 「主控附言」承载调度层信息：state.md 备忘中与本任务相关的提醒、中断残留警告（"touches 下可能有上次中断的半成品"）、临时资源约束。不得篡改验收标准、不得替 Worker 预设实现方案。
- 「上次失败的反思」是主控亲笔的判断性内容，禁止无反思的原样重发。

## 回执格式：`loop/reports/<task-id>.md`

由 Worker 在结束前写出（无论成败）。这是 Worker 留给世界的唯一遗产——它的上下文即将销毁，一切有价值的信息必须写在这里。模板：

```markdown
# 回执: <task-id>
- 结果: 完成 | 未完成
- 自查: <逐条对照验收标准, 每条标注 ✅/❌ 及一句话证据>

## 产出物
<创建/修改的文件列表, 每个文件一句话说明>

## 关键决策
<执行中做出的、会影响后续任务的技术选择, 每条一行。
 例如: "日期格式统一用 ISO 8601"、"数据存储选了 JSON 而非 SQLite, 因为..."。
 没有则写"无">

## 对下游的提醒
<下游任务使用我的产出时需要知道的事: 接口约定、已知限制、坑。没有则写"无">

## 遗留问题  <!-- 未完成或部分完成时必填 -->
<什么没做完、卡在哪里、缺什么信息或权限>
```

**回执的上下文纪律**：回执全文必须 ≤60 行——**主控只读前 60 行，超出部分对主控不存在**（预算由读者端强制执行）。它是摘要不是日志：主控永不读产出物全文，回执写不清楚的信息对主控就是不存在的。

## 验证简报格式：`loop/briefs/<task-id>.verify.md`

由主控在派发 Verifier 前生成。同样是指针式。模板：

```markdown
# 验证简报: <task-id> <任务标题>

## 验证依据（按序读取）
1. TASKS.md：「全局上下文」节（通用约束）+ 「[<task-id>]」任务节
   （acceptance 逐条核查; constraints 与 touches 同为核查对象）。其余任务节是禁区。
2. 回执 loop/reports/<task-id>.md：只读「产出物」与「关键决策」两节
   （用于圈定核查范围与重点部位）。不要参照其自查结论——你的价值在于独立重查。

## 禁读清单（评判独立性红线）
- loop/lessons.md 与 loop/decisions.md（经验与决策只影响提案, 不得影响评判）
- loop/briefs/<task-id>*.md（任务简报, 含失败叙事）与 loop/verdicts/ 下任何历史裁决

## 授权验证命令
- acceptance 中出现的构建/测试/grep 等机械验证命令, 视为已授权
- <主控额外授权的读操作, 如起本地 dev server 查看页面; 无则删除本行>

## 附加约束  <!-- 主控调度层补充, 如"本次为复核, 构建前先 clean"; 无则写"无" -->
无
```

## 裁决格式：`loop/verdicts/<task-id>.md`

由 Verifier 在结束前写出。模板：

```markdown
# 裁决: <task-id>
- 总裁决: 通过 | 不通过

## 逐条核查
<每条验收标准与约束一行: ✅/❌/⚠️(无法验证) + 一句话证据。
 ❌ 条目必须写明: 哪个文件/哪个场景、观察到什么、与预期的差异>

## 修复建议  <!-- 仅不通过时必填 -->
<2-3 行: 基于你读过的产出物全文, 指出问题根源与建议的修复方向。
 你是唯一读过全文的角色, 主控将以此为主要依据撰写重试反思>

## 备注  <!-- 可省略 -->
<⚠️ 条目的原因、核查中发现的验收标准之外的隐患>
```

**裁决的上下文纪律**：裁决全文应控制在 **30 行以内**。存在任一 ❌ 时总裁决必须为"不通过"；⚠️ 不导致不通过，但主控须记入日志。单任务 ⚠️ 条目超过标准总数一半时，主控不得静默标记 done，应升级询问用户。

## 验收流程（主控侧）

1. 读回执（只读前 60 行）。回执缺失或格式残缺 → 视同验收失败。
2. 回执报告任务超出单次上下文可完成范围 → 立即升级（见「重试与升级规则」），不派 Verifier。
3. **越界检查**：比对回执「产出物」列表与该任务 touches——出现范围外文件即视同验收失败（越界写入破坏并行安全，须在反思中明确指出）。**清单笔误特例**：若 touches 中某路径在项目中不存在，且回执产出物路径与任务目标明显对应（典型：清单写错目录名），主控**不修改 TASKS.md**，而是在 state.md 备忘登记「Tn touches 笔误：清单写 X，实际为 Y」，本次验收与后续并行判定按实际路径执行，并在 FINAL 遗留问题中提请人类修正清单。其余越界一律判失败。
4. 核对自查表：Worker 自报有 ❌ → 直接失败，跳到重试（省去一次验证调用）。
5. **独立验证**：写指针式验证简报，派发 Verifier（指令："读 `prompts/verifier.md` 并遵守其中规则，然后执行 `loop/briefs/<id>.verify.md` 描述的验证"），读回裁决。**验证互斥**：共享 `resources` 标签的任务，其 Verifier 不得与同标签任务的 Worker 或 Verifier 同时在途——资源占用从派发 Worker 起持续到读回裁决止（见 task-list-spec）。
6. 裁决"通过" → 标记 `done`，并把回执「关键决策」追加进 `loop/decisions.md`；"不通过" → 标记 `failed` 并进入重试，「上次失败的反思」以裁决的 ❌ 条目与「修复建议」为主要依据。

## 可疑裁决复核（主控的唯一仲裁手段）

出现下列任一情形：

- 裁决与回执自查直接矛盾，且 ❌ 证据指向**环境噪音**（并行构建残留、缓存、锁竞争）而非产出物本身；
- 同轮多份裁决对同一全局事实（如"项目能否编译通过"）结论相反；
- 裁决缺失或格式残缺。

处理：重派一名**全新** Verifier 复核，验证简报「附加约束」注明"本次为复核，构建/测试前先执行 clean"，以复核结论为准，**不消耗任务重试次数**。主控在任何情况下不得以亲自读产出物、亲自跑命令的方式仲裁——实测一次亲自仲裁烧掉的预算约等于三个任务的正常调度，且破坏铁律。复核结论仍可疑 → 升级用户。

## 全局决策记录 `loop/decisions.md`

跨任务传递信息的补充通道，解决**传递性依赖的信息丢失**（直接依赖注入只覆盖一跳，链条更长的决策会丢）。规则：

- 仅主控在任务验收通过时追加写入；格式：`- [T1] 存储键名统一以 "bm:" 为前缀`。
- Worker 按简报阅读清单自行读取全文。
- **只追加，禁止顺手改写既有条目**。上限 **40 行**：仅当超限时做一次合并同类项、删除已失效条目的压缩（压缩记日志一条）——频繁重写是预算泄漏。

## 重试与升级规则

- 每个任务最多重试 **2 次**（共 3 次尝试；上限可由元循环在 `protocol/meta-spec.md` 白名单内调整, 当前值见 state.md 运行配置）。**例外**：Worker 自查诚实报 ❌ 导致的失败，首次不计入重试次数（它省掉了一次验证调用；诚实自查不应在机制上受罚）——但同一任务此例外只适用一次。可疑裁决复核同样不消耗重试次数。
- 重试前主控必须写出「上次失败的反思」并放入新简报：失败在哪条标准、推测原因（以裁决「修复建议」为主要依据）、这次要怎么改。禁止无反思的原样重发。
- 重试前先归档上次尝试的简报/回执/裁决（见文首归档规则）。
- 重试仍失败 → 标记 `blocked`，继续调度其他不受影响的任务；所有可做任务做完后，向用户汇总全部 blocked 任务及其失败史（依据归档的历次回执与裁决），请求人工介入。
- **立即升级**（不消耗重试次数）的情形：任务需要清单未提供的凭据/权限、发现清单内在矛盾、Worker 报告需要破坏性操作、Worker 报告任务超出单次上下文可完成范围（应由用户拆分任务，而非让 Worker 硬撑）。
