# 主从交接协议（handoff-spec）

主控、Worker 与 Verifier 之间**只通过三种文件**通信：简报（brief，主控→Worker）、回执（report，Worker→主控）、裁决（verdict，Verifier→主控；其输入是主控写的验证简报）。禁止任何隐式信息传递（如"Worker 应该记得上一轮……"——Worker 没有上一轮，每个 Worker、每个 Verifier 都是全新上下文）。

**重试归档规则**：三类文件都按 `<task-id>.md` 命名，指向**当前尝试**。派发重试前，主控必须先把上一次尝试的简报、回执、裁决重命名归档为 `<task-id>.attempt<n>.md`（n 为上次的尝试次数），再写新简报。失败历史因此完整保留——blocked 升级时的失败史汇总、反思的撰写，都以归档文件为依据。

## 简报格式：`loop/briefs/<task-id>.md`

由主控在派发前生成。模板：

```markdown
# 任务简报: <task-id> <任务标题>
- 尝试次数: 1  <!-- 重试时递增 -->

## 全局上下文
<从 TASKS.md「全局上下文」一节原样复制>

## 任务目标
<从 TASKS.md 该任务的正文原样复制>

## 开工前必读
<context_files 列表；无则写"无">

## 上游交接
<对每个 depends_on 的任务, 从其回执中摘录「关键决策」和「对下游的提醒」两节;
 无依赖则写"无">

## 全局决策记录
<loop/decisions.md 全文原样注入; 文件不存在或为空则写"无"。
 这是传递性依赖信息的通道: 直接依赖之外的上游决策也在其中>

## 允许写入的路径
<touches 列表。此范围之外只读, 不得创建或修改任何文件>

## 验收标准
<acceptance 列表原样复制。你交付前必须逐条自查>

## 约束
<constraints 列表 + 主控认为需要补充的约束; 无则写"无">

## 上次失败的反思  <!-- 仅重试时包含此节 -->
<主控写明: 上次验收未通过的具体标准、失败原因分析、本次应改变的做法>
```

**主控生成简报的纪律**：简报内容来自 TASKS.md 和上游回执的**机械组装**，主控可以补充调度层面的提醒，但不得篡改验收标准、不得替 Worker 预设实现方案。

## 回执格式：`loop/reports/<task-id>.md`

由 Worker 在结束前写出（无论成败）。这是 Worker 留给世界的唯一遗产——它的上下文即将销毁，一切有价值的信息必须写在这里。模板：

```markdown
# 回执: <task-id>
- 结果: 完成 | 未完成
- 自查: <逐条对照验收标准, 每条标注 ✅/❌ 及一句话证据>

## 产出物
<创建/修改的文件列表, 每个文件一句话说明>

## 关键决策
<执行中做出的、会影响后续任务的技术选择, 每条一行。
 例如: "日期格式统一用 ISO 8601"、"数据存储选了 JSON 而非 SQLite, 因为..."。
 没有则写"无">

## 对下游的提醒
<下游任务使用我的产出时需要知道的事: 接口约定、已知限制、坑。没有则写"无">

## 遗留问题  <!-- 未完成或部分完成时必填 -->
<什么没做完、卡在哪里、缺什么信息或权限>
```

**回执的上下文纪律**：回执全文应控制在 **60 行以内**。它是摘要不是日志——主控只读回执、永不读产出物全文，回执写不清楚的信息对主控就是不存在的。

## 验证简报格式：`loop/briefs/<task-id>.verify.md`

由主控在派发 Verifier 前生成。模板：

```markdown
# 验证简报: <task-id> <任务标题>

## 验收标准
<acceptance 列表原样复制。逐条核查>

## 约束
<constraints 列表原样复制 (含全局上下文中的通用约束); 无则写"无"。
 约束违反同样导致不通过>

## 产出物范围
<该任务的 touches 列表 + 回执「产出物」一节摘录。核查限于此范围;
 若回执产出物中出现 touches 之外的文件, 属越界写入, 直接判不通过>

## Worker 回执要点
<从回执摘录「关键决策」, 帮助 Verifier 定位重点核查部位>

## 允许执行的验证命令
<主控从 acceptance 中提取的机械验证命令 (构建/测试/grep 等);
 需要起本地服务查看页面等读操作时在此明示授权; 无则写"无">
```

## 裁决格式：`loop/verdicts/<task-id>.md`

由 Verifier 在结束前写出。模板：

```markdown
# 裁决: <task-id>
- 总裁决: 通过 | 不通过

## 逐条核查
<每条验收标准与约束一行: ✅/❌/⚠️(无法验证) + 一句话证据。
 ❌ 条目必须写明: 哪个文件/哪个场景、观察到什么、与预期的差异>

## 修复建议  <!-- 仅不通过时必填 -->
<2-3 行: 基于你读过的产出物全文, 指出问题根源与建议的修复方向。
 你是唯一读过全文的角色, 主控将以此为主要依据撰写重试反思>

## 备注  <!-- 可省略 -->
<⚠️ 条目的原因、核查中发现的验收标准之外的隐患>
```

**裁决的上下文纪律**：裁决全文应控制在 **30 行以内**。存在任一 ❌ 时总裁决必须为"不通过"；⚠️ 不导致不通过，但主控须记入日志。单任务 ⚠️ 条目超过标准总数一半时，主控不得静默标记 done，应升级询问用户。

## 验收流程（主控侧）

1. 读回执。回执缺失或格式残缺 → 视同验收失败。
2. 回执报告任务超出单次上下文可完成范围 → 立即升级（见下节），不派 Verifier。
3. **越界检查**：比对回执「产出物」列表与该任务 touches——出现范围外文件即视同验收失败（越界写入破坏并行安全，须在反思中明确指出）。
4. 核对自查表：Worker 自报有 ❌ → 直接失败，跳到重试（省去一次验证调用）。
5. **独立验证**：生成验证简报，派发 Verifier（指令："读 `prompts/verifier.md` 并遵守其中规则，然后执行 `loop/briefs/<id>.verify.md` 描述的验证"），读回裁决。主控不亲自跑任何验证命令——验证过程的海量输出由 Verifier 消化，主控只读 ≤30 行的裁决。
6. 裁决"通过" → 标记 `done`，并把回执「关键决策」追加进 `loop/decisions.md`（每条一行，前缀 `[<task-id>]`；无决策则不追加）；"不通过" → 标记 `failed` 并进入重试，「上次失败的反思」以裁决的 ❌ 条目与「修复建议」为主要依据。裁决缺失或格式残缺 → 重派一次 Verifier，仍失败则视同验收失败。

## 全局决策记录 `loop/decisions.md`

跨任务传递信息的补充通道，解决**传递性依赖的信息丢失**（直接依赖注入只覆盖一跳，链条更长的决策会丢）。规则：

- 仅主控在任务验收通过时追加写入；格式：`- [T1] 存储键名统一以 "bm:" 为前缀`。
- 全文注入每份任务简报的「全局决策记录」一节。
- 上限 **40 行**：超限时主控合并同类项、删除已失效条目（压缩是主控的职责，日志中记录一条）。

## 重试与升级规则

- 每个任务最多重试 **2 次**（共 3 次尝试）。**例外**：Worker 自查诚实报 ❌ 导致的失败，首次不计入重试次数（它省掉了一次验证调用；诚实自查不应在机制上受罚）——但同一任务此例外只适用一次。
- 重试前主控必须写出「上次失败的反思」并放入新简报：失败在哪条标准、推测原因（以裁决「修复建议」为主要依据）、这次要怎么改。禁止无反思的原样重发。
- 重试前先归档上次尝试的简报/回执/裁决（见文首归档规则）。
- 重试仍失败 → 标记 `blocked`，继续调度其他不受影响的任务；所有可做任务做完后，向用户汇总全部 blocked 任务及其失败史（依据归档的历次回执与裁决），请求人工介入。
- **立即升级**（不消耗重试次数）的情形：任务需要清单未提供的凭据/权限、发现清单内在矛盾、Worker 报告需要破坏性操作、Worker 报告任务超出单次上下文可完成范围（应由用户拆分任务，而非让 Worker 硬撑）。
