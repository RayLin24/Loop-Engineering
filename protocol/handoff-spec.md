# 主从交接协议（handoff-spec）

主控、Worker 与 Verifier 之间**只通过文件**通信：简报（主控→Worker）、回执（Worker→主控）、裁决（Verifier→主控）。禁止任何隐式信息传递——每个 Worker、每个 Verifier、甚至**每个主控 tick** 都是全新上下文，没有"上一轮的记忆"。

**命名规则**：交接文件按 `<task-id>.attempt<n>.md` 命名（n 从 1 起）。每次尝试的文件天然独立，**无需归档重命名**——失败历史自动完整保留，blocked 升级时的失败史汇总、重试反思的撰写，直接引用历次文件。

## 事务性工作区（每次尝试 = 一个事务）

每次任务尝试在**隔离的 git worktree** 中执行，验收通过才合并主干。这是并行安全、失败回滚、超时重派安全的共同基础。

```
派发:   git worktree add loop/worktrees/<task-id>.attempt<n> -b loop/<task-id>-a<n>
执行:   Worker 只在该 worktree 内工作, 结束时 git commit（含失败/未完成, 便于验尸）
验证:   Verifier 在该 worktree 内跑 checks 与核查
通过:   主控将该分支合并回主干（squash 合并, 提交信息 "loop: <task-id> <标题>"）,
        删除 worktree 与分支, 追加 landed 事件
不通过: 删除 worktree 与分支, 追加 discarded 事件——主干零污染, 回滚零成本
```

规则：

- **失败零污染**：重试 Worker 面对的永远是干净主干 + 反思，不存在"检查半成品"。上次尝试的产出如需参考，主控可在反思中摘录要点（来自裁决），但不复用脏工作区。
- **合并冲突 = 并行冲突的事后检测**：`touches` 是调度提示（尽量避免冲突），不是安全保证。若合并时发生冲突（说明 touches 声明不准，或超时重派出现了双尝试），**丢弃后到者**（`discarded`, reason: "merge-conflict"），该任务按失败重试（`class: "transient"`, 不消耗重试），并在 state.md 备注提醒人类修正 touches。禁止主控手工解决冲突——那等于亲自执行任务。
- **非 git 项目降级**：无 git 时退化为直接在主干执行 + 失败时靠重试 Worker 自行清理，同时并行度强制为 1。协议其余部分不变。框架强烈建议 `git init`（一条命令的代价，换来全部事务能力）。

## 验收即代码：checks 脚本

每个任务的验收标准分两层处理：

- **机械层** → `loop/checks/<task-id>.sh`：可脚本化的标准（文件存在、grep 内容、构建、测试命令）。由**首次验证该任务的 Verifier 顺手编写**（它反正要逐条执行这些检查，写成脚本是同一份工作的可复用形态）。脚本要求：任何一条标准不达标即非零退出并打印哪条失败；幂等可重复执行；只读（不修改产出物）。
- **语义层** → 留给 Verifier 人工核查：布局正常、逻辑合理、文案完整等无法脚本化的标准。

checks 脚本带来三个能力：

1. **重试验证免费**：重试尝试直接跑现成脚本，Verifier 只需复核语义层与修脚本（若标准理解有误）。
2. **回归保护**：任务落地后，主控让 Verifier（或直接机械执行）重跑**所有 done 任务中 touches 与本次变更相交**的 checks 脚本。失败 → 追加 `regression` 事件，生成一个修复性重试派发给肇事任务（reason 注明破坏了哪个任务的哪条标准）。`done` 由此从"验收过一次"升级为"当前仍然成立"。
3. **交付物**：循环结束时 `loop/checks/` 是项目验收测试套件的雏形，随 FINAL.md 交付。

## 简报格式：`loop/briefs/<task-id>.attempt<n>.md`

由主控在派发前**机械组装**（内容来自 TASKS.md 与上游回执，主控可补充调度层提醒，但不得篡改验收标准、不得替 Worker 预设实现方案）：

```markdown
# 任务简报: <task-id> <任务标题>
- 尝试次数: <n>
- 工作区: loop/worktrees/<task-id>.attempt<n>   <!-- 你的一切读写都在这里 -->

## 全局上下文
<从 TASKS.md「全局上下文」原样复制>

## 任务目标
<从 TASKS.md 该任务正文原样复制>

## 开工前必读
<context_files 列表; 无则"无">

## 上游交接
<对每个 depends_on 任务, 从其回执摘录「关键决策」「对下游的提醒」; 契约文件列出路径即可>

## 相关全局决策
<从决策索引中筛选与本任务相关的条目注入（按 scope 标签与 touches/依赖匹配）,
 而非全文注入; 无则"无">

## 允许写入的路径
<touches 列表。工作区内此范围之外只读>

## 验收标准
<acceptance 原样复制。交付前逐条自查>

## 约束
<constraints + 主控补充; 无则"无">

## 已产生的外部副作用   <!-- 仅重试且上次回执报告过副作用时包含 -->
<上次尝试已执行的不可回滚外部操作（调用过的 API、发过的通知等), 本次勿重复>

## 上次失败的反思   <!-- 仅重试时包含 -->
<未通过的具体标准、失败原因分析（以裁决 ❌ 条目与修复建议为依据）、本次应改变的做法>
```

## 回执格式：`loop/reports/<task-id>.attempt<n>.md`

由 Worker 结束前写出（**无论成败**）。≤60 行——主控只读回执、永不读产出物全文：

```markdown
# 回执: <task-id> (尝试 <n>)
- 结果: 完成 | 未完成
- 自查: <逐条对照验收标准, ✅/❌ + 一句话证据>

## 产出物
<工作区内创建/修改的文件列表, 每个一句话>

## 关键决策
<影响后续任务的技术选择, 每条一行, 建议带 scope 标签,
 如 "- [naming] 存储键名统一 bm: 前缀"。没有则"无">

## 对下游的提醒
<接口约定、已知限制、坑。契约文件（schema/类型定义）写路径。没有则"无">

## 外部副作用   <!-- 有 effects 声明的任务必填 -->
<本次已实际产生的不可回滚外部操作清单; 无则"无">

## 遗留问题   <!-- 未完成时必填 -->
<什么没做完、卡在哪、缺什么信息/权限>
```

## 验证简报与裁决

验证简报 `loop/briefs/<task-id>.attempt<n>.verify.md`（主控生成）在原有各节（验收标准、约束、产出物范围、回执要点、允许执行的验证命令）之外增加：

```markdown
## checks 脚本
<attempt 1: "loop/checks/<task-id>.sh 不存在, 请按本简报的机械验证项编写它并执行";
 attempt ≥2: "先执行 loop/checks/<task-id>.sh, 再做语义核查; 若脚本与验收标准不符可修正脚本">

## 回归检查   <!-- 仅落地后回归验证时使用的简报包含 -->
<需重跑的相关任务 checks 脚本列表>
```

裁决 `loop/verdicts/<task-id>.attempt<n>.md`（Verifier 写出，≤30 行）：

```markdown
# 裁决: <task-id> (尝试 <n>)
- 总裁决: 通过 | 不通过
- 失败分类: transient | deterministic | environmental   <!-- 仅不通过时 -->

## 逐条核查
<每条标准与约束: ✅/❌/⚠️(无法验证) + 一句话证据。机械项可合并为
 "checks 脚本 N/N 通过"。❌ 必须写明文件/场景/观察到什么/与预期差异>

## 修复建议   <!-- 仅不通过时必填, 2-3 行 -->
<问题根源与修复方向。你是唯一读过产出物全文的角色, 重试反思全靠这节>

## 备注   <!-- 可省略 -->
<⚠️ 原因、标准之外的隐患>
```

存在任一 ❌ → 总裁决必须"不通过"。⚠️ 不导致不通过但须记录；单任务 ⚠️ 超过标准总数一半 → 主控不得静默 done，写入 `loop/questions/` 升级（见下文，循环不停）。

## 失败分类学（决定重试策略）

| class | 判定 | 处理 |
|---|---|---|
| `transient` 瞬时 | 网络抖动、限流、超时中断、合并冲突 | 直接重试，**不写反思、不消耗重试次数**；同一任务连续 3 次 transient → 升格 blocked |
| `deterministic` 确定性 | 逻辑错误、标准未达、约束违反 | 写反思重试，消耗重试次数（上限 2 次） |
| `environmental` 环境性 | 缺凭据/权限、清单矛盾、需破坏性操作、任务超出单次上下文 | **不重试**，直接写入 `loop/questions/` 等待人类，任务标 `waiting_human` |

分类来源：Verifier 裁决的 `失败分类` 字段；Worker 回执"未完成 + 遗留问题"时由主控依据遗留问题内容判定。

**诚实自查激励保留**：Worker 自查诚实报 ❌ 的首次失败（deterministic）不消耗重试次数（省掉一次验证调用），同一任务此例外仅一次。

## 外部副作用与幂等（effects 声明的任务）

任务清单声明了 `effects`（如 `network`、`db-write`、`deploy`）的任务：

- Worker 执行不可回滚的外部操作前，尽量使用幂等键（简报中的尝试编号可作请求 ID）；回执**必须**如实填写「外部副作用」节。
- 重试简报注入上次的副作用清单，Worker 据此跳过已完成的外部操作——避免重复扣款/重复发布类事故。
- worktree 只隔离文件系统，**隔离不了外部世界**——这就是副作用要走日志而非事务的原因。

## 异步人机通道（升级不再停机）

### `loop/questions/<task-id>.md` — 主控问人类

需要人类决策时（environmental 失败、⚠️ 过半、清单矛盾），主控写问题文件并将任务标 `waiting_human`，**循环继续推进其他任务**：

```markdown
# 问题: <task-id> <一句话问题>
- 状态: 待回答
## 背景
<2-5 行: 发生了什么, 为什么需要人类>
## 选项
<可给出建议选项>
## 回答   <!-- 人类在此作答, 并把状态改为"已回答" -->
```

每 tick 开头扫描：状态"已回答" → 追加 `task_reset` 事件，答案摘录进该任务下次简报。

### `loop/inbox/` — 人类指令主控

人类随时投递指令文件（如 `pause.md`、`skip-T5.md`、内容自由书写），每 tick 开头消费：执行指令 → 追加 `note` 事件记录 → 删除或标记指令文件。修订 TASKS.md 无需通过 inbox（每 tick 自动重读同步，见 event-spec）。

## 验收流程（主控 tick 内，对每个有回执的 dispatched 任务）

1. 读回执。缺失/残缺且租约过期 → 按中断处理（见 event-spec 租约节）。
2. 回执"未完成" → 按遗留问题判定失败分类，走对应策略；报告超出单次上下文 → environmental，写 questions 建议人类拆分。
3. **越界检查**：产出物超出 touches → deterministic 失败（越界破坏并行调度，反思中明确指出）。
4. 自查表含 ❌ → 直接 deterministic 失败，**不派 Verifier**（省一次验证；诚实自查首次不耗重试）。
5. 否则生成验证简报、派 Verifier、读裁决。裁决缺失/残缺 → 重派一次 Verifier，再失败按 transient 处理。
6. 通过 → 合并 worktree 落地（`landed`）、追加 `decision` 事件、触发回归 checks、`task_done`。不通过 → 丢弃 worktree（`discarded`）、按失败分类处理。
