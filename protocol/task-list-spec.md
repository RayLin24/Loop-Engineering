# 任务清单规范（task-list-spec）

`TASKS.md` 是整个循环的**唯一输入**，由人类编写。主控和 Worker 都**不得修改**此文件；但人类可以**随时**修订它——主控每 tick 开头会重读并同步（同步规则见 [event-spec.md](event-spec.md) 的修订协议），无需等循环暂停。

## 文件结构

```markdown
# 任务清单：<项目名>

## 全局上下文
<项目背景、技术栈、通用约束。这一节会原样注入到每个任务简报中，
 请控制在 30 行以内，只写所有任务都需要知道的信息。>

## 任务

### [T1] <任务标题>
```yaml
id: T1
depends_on: []
touches:
  - src/data/
acceptance:
  - <可机器/肉眼验证的标准 1>
  - <可验证标准 2>
context_files:
  - docs/requirements.md
constraints:
  - <本任务特有的约束>
```
<正文：用自然语言描述任务目标与背景，写给一个
 "对项目一无所知、只能看到本简报"的执行者。>

### [T2] <下一个任务>
...
```

## 字段定义

| 字段 | 必填 | 说明 |
|---|---|---|
| `id` | 是 | 任务唯一标识，格式 `T<数字>`，与标题中 `[T1]` 一致 |
| `depends_on` | 是 | 依赖的任务 id 列表，空列表 `[]` 表示无依赖 |
| `touches` | 是 | 本任务允许**写入**的文件/目录路径列表（并行调度提示；实际冲突由合并时检出，见 handoff-spec 事务规则） |
| `acceptance` | 是 | 验收标准列表，Worker 据此自查，Verifier 据此核查并编译 checks 脚本 |
| `context_files` | 否 | Worker 开工前必须阅读的已有文件 |
| `constraints` | 否 | 任务级约束（禁区、风格要求、性能要求等） |
| `resources` | 否 | 本任务独占的共享资源标签列表（如 `port:3000`、`npm-install`、`db:dev`），并行调度判定依据之二 |
| `timeout` | 否 | 单次尝试的时间上限（租约时长），如 `90m`。默认 `30m`。超时视为挂死中断，丢弃工作区重派（不消耗重试次数），机制见 event-spec 租约节 |
| `effects` | 否 | 本任务会产生的**不可回滚外部副作用**标签列表（如 `network`、`db-write`、`deploy`、`email`）。声明后 Worker 必须在回执记录已产生的副作用，重试时注入避免重复执行（见 handoff-spec 幂等节） |
| `risk` | 否 | 标注 `high` 的任务（安全、支付、数据迁移等）验收时派 2 个独立 Verifier 从不同视角核查（正确性视角 + 安全/回归视角），任一不通过即不通过 |

## 编写规则

1. **验收标准必须可验证，且尽量可机械化**。坏例子："代码质量良好"；好例子："`node --test` 全部通过"、"`index.html` 在浏览器打开无 console 报错"、"文件 `api.md` 包含全部 4 个接口的请求/响应示例"。可机械化的标准（命令、文件存在性、grep 可查的内容）会被编译成 `loop/checks/<id>.sh` 脚本——重试免费复验、落地后回归保护、结束后作为验收测试套件交付；纯语义标准（"布局美观"）每次都要消耗一次 LLM 核查且无回归保护。**能写成命令的就写成命令。**
2. **`touches` 决定并行调度**。两个就绪任务的 `touches` 互不重叠（且无依赖关系）时会被并行派发。写得越精确，并行度越高。写错也不会静默损坏——每次尝试在隔离工作区执行，真实冲突会在合并时显式暴露（后到者丢弃重做）——但频繁冲突浪费重试，touches 仍应尽量写准。写成项目根目录会使并行度归零，主控启动时会提醒收窄。
3. **文件不重叠 ≠ 无冲突**。同占一个端口、同时 `npm install` 写 lockfile、同写一个数据库——这类共享资源冲突用 `resources` 声明（同名标签不同批并行）。并行判定 = touches 与 resources **均**无重叠。
4. **依赖必须无环**。主控启动时做环检测，发现环直接升级用户。
5. **任务粒度**：一个任务 = 一个 Worker 一次上下文内可完成的工作量。经验值：产出 1–5 个文件、或一个内聚的功能模块。太大的任务在编写清单时就拆开。运行时 Worker 报告"超出单次上下文"会转为问题文件请你拆分。
6. **正文自包含**：Worker 看不到其他任务的正文，只能看到依赖任务回执中的摘要。每个任务正文要独立可读。
7. **敏感信息写引用，不写明文**。「全局上下文」会注入每个简报，凭据/密钥请写引用方式（环境变量名、配置文件路径，如"数据库密码见 `.env` 的 `DB_PASSWORD`"）。
8. **`context_files` 列具体文件，不列目录**。确实需要了解一个目录时，指出其中最有代表性的 1–3 个文件。
9. **`timeout` 不必精确估计**。默认 30 分钟适合多数任务；只有确知会长时间运行（完整测试套件、大构建）的任务才需要放宽。宁可默认宽松，超时的代价只是一次免费重派。
10. **有外部副作用的任务必须声明 `effects`**。这是防止重试时重复扣款/重复发布的唯一防线——文件系统的失败可以靠事务工作区回滚，外部世界不能。

## 依赖语义

- `depends_on: [T1, T2]`：T1 和 T2 都 `done` 后本任务才就绪。
- 依赖任务回执的「关键决策」「对下游的提醒」由主控自动注入本任务简报——任务间传递信息的**主通道**。需要跨任务传递的信息，务必让上游写进回执。
- **接口契约走产物**：上游产出的机器可读契约（JSON Schema、`.d.ts`、接口文档）比回执转述更可靠——让上游任务的 acceptance 要求产出契约文件，下游任务将其列入 `context_files`，Verifier 还可核查下游是否符合契约。
- 补充通道：已验收任务的「关键决策」进入决策索引（`decision` 事件，带 scope 标签），主控组装简报时**筛选相关条目**注入（而非全文），保证全局性约定（命名规则、数据格式）不因依赖链过长而丢失。

## 「审查即任务」模式（需要独立 code review 时）

框架不设固定 Reviewer 角色——每个任务已由独立上下文的 Verifier 核查。若还需要更主观的代码审查（可维护性、架构一致性、安全隐患），把审查写成普通任务即可：派出去的 Worker 天然是不带实现者偏见的干净上下文 reviewer。

```yaml
id: T9
depends_on: [T1]          # 审查 T1 的产出
touches:
  - docs/reviews/T1-review.md   # 只允许写审查报告, 被审代码全部只读
acceptance:
  - docs/reviews/T1-review.md 存在, 包含「问题清单」与「改进建议」两节
  - 问题清单中每条注明 文件:行号 与严重程度 (高/中/低)
```

审查发现的问题要转化为行动时，写进后续修复任务的正文（或由用户追加新任务），而不是让 reviewer 直接改代码。对安全敏感的任务，更轻量的替代是直接标 `risk: high` 启用双 Verifier。
