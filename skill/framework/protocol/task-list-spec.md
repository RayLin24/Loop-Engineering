# 任务清单规范（task-list-spec）

`TASKS.md` 是整个循环的**唯一输入**，由人类编写。循环运行期间主控和 Worker 都**不得修改**此文件（任务状态由 `loop/events.jsonl` 管理，见 [event-spec.md](event-spec.md)）；人类可随时修订，每轮开头自动同步（修订协议见 event-spec）。

## 文件结构

```markdown
# 任务清单：<项目名>

## 全局上下文
<项目背景、技术栈、通用约束。这一节在每个任务/验证简报的阅读清单中，
 Worker 与 Verifier 都会读取，请控制在 30 行以内，只写所有任务都需要知道的信息。>

## 任务

### [T1] <任务标题>
```yaml
id: T1
depends_on: []
touches:
  - src/data/
acceptance:
  - <可机器/肉眼验证的标准 1>
  - <可验证标准 2>
context_files:
  - docs/requirements.md
constraints:
  - <本任务特有的约束>
```
<正文：用自然语言描述任务目标与背景，写给一个
 "对项目一无所知、只能看到本简报"的执行者。>

### [T2] <下一个任务>
...
```

## 字段定义

| 字段 | 必填 | 说明 |
|---|---|---|
| `id` | 是 | 任务唯一标识，格式 `T<数字>`，与标题中 `[T1]` 一致 |
| `depends_on` | 是 | 依赖的任务 id 列表，空列表 `[]` 表示无依赖 |
| `touches` | 是 | 本任务允许**写入**的文件/目录路径列表（并行安全判定与合并前越界检查的依据） |
| `acceptance` | 是 | 验收标准列表，Verifier 据此核查，Worker 据此自查 |
| `context_files` | 否 | Worker 开工前必须阅读的已有文件 |
| `constraints` | 否 | 任务级约束（禁区、风格要求、性能要求等） |
| `resources` | 否 | 本任务独占的共享资源标签列表（如 `port:3000`、`npm-install`、`db:dev`），并行安全判定依据之二 |
| `timeout` | 否 | 本任务的租约时长（如 `45m`、`2h`），默认 30m；超时按中断处理（见 event-spec 租约） |
| `risk` | 否 | `high` 表示高风险任务：验证强度强制为"严格"（Verifier 通读全部产出物），不受运行配置调低影响 |
| `effects` | 否 | 本任务会产生的**外部副作用**列表（部署、写数据库、调用付费 API 等不可回滚操作）。声明后：Verifier 禁止触发这些副作用；无人值守运行前人类应逐条确认 |

## 编写规则

1. **验收标准必须可验证**。坏例子："代码质量良好"；好例子："`node --test` 全部通过"、"`index.html` 在浏览器打开无 console 报错"、"文件 `api.md` 包含全部 4 个接口的请求/响应示例"。
2. **机械可验证的标准单独成行**。能写成命令、文件存在性、grep 断言的标准，措辞上就写成可执行形式（"`node --test` 全部通过"而非"测试没问题"）——任务落地后这些条目会被编译成 `loop/checks/<id>.sh` 回归脚本，在后续每次落地时重跑，保护已验收成果不被破坏（见 handoff-spec 工作区事务）。写法越机械，回归网越密。
3. **`touches` 决定并行与越界**。两个就绪任务的 `touches` 路径互不重叠（且无依赖关系）时，主控可以并行派发它们。写得越精确，并行度越高；不确定就写宽一点（牺牲并行换安全）。注意：把 `touches` 写成项目根目录会使它与所有任务重叠，并行度归零、全部串行——主控启动时检测到这种情况会提醒你收窄。路径写法规范：目录以 `/` 结尾，文件写全路径；重叠判定按前缀包含关系（`src/js/` 与 `src/js/app.js` 重叠）。
4. **文件不重叠 ≠ 无冲突**。两个任务可能都要占同一端口起 dev server、同时 `npm install` 写 lockfile、连同一个数据库——这类共享资源冲突用 `resources` 字段声明（同名标签的任务不会同批并行）。主控的并行判定是 touches 与 resources **均**无重叠。**互斥覆盖验证阶段**：资源占用从派发 Worker 起持续到读回裁决止——同标签任务的 Verifier 之间、Verifier 与 Worker 之间同样不并行。（构建产物目录不再需要 resources 声明：每次尝试在独立 git worktree 中执行，产物天然隔离；resources 只用于端口、数据库、lockfile 这类真共享资源。）
5. **依赖必须无环**。主控启动时会做环检测，发现环会直接升级给用户。
6. **任务粒度**：一个任务 = 一个 Worker 一次上下文内可完成的工作量。经验值：产出 1–5 个文件、或一个内聚的功能模块。太大的任务应在编写清单时就拆开，而不是指望 Worker 自己拆。耗时明显超过 30 分钟的任务应拆小，或显式声明 `timeout`。
7. **正文自包含**：Worker 看不到其他任务的正文，只能看到依赖任务回执中的摘要。所以每个任务正文要独立可读。
8. **敏感信息写引用，不写明文**。「全局上下文」会被每个任务的 Worker 与 Verifier 读取，凭据/密钥/token 请写引用方式（环境变量名、配置文件路径，如"数据库密码见 `.env` 的 `DB_PASSWORD`"），避免明文扩散到所有子代理上下文。
9. **`context_files` 列具体文件，不列目录**。写整个目录会让 Worker 在勘察阶段吞掉大量无关内容，挤占真正干活的上下文。确实需要 Worker 了解一个目录时，指出其中最有代表性的 1–3 个文件。

## 依赖语义

- `depends_on: [T1, T2]`：T1 和 T2 都 `done` 后本任务才就绪。
- 依赖任务的回执中「关键决策」和「对下游的提醒」两节，会被列入本任务简报的阅读清单、由 Worker 开工时自行读取——这是任务间传递信息的**主通道**。需要跨任务传递的信息，务必让上游任务写进回执。
- 补充通道：所有已验收任务的「关键决策」汇入 `loop/decisions.md`（决策索引投影），Worker 按简报给出的 scope 列表匹配读取（`global` + 相关 scope），保证非直接依赖的全局性决策不因依赖链过长而丢失，同时避免全文注入挤占 Worker 上下文。

## 「审查即任务」模式（需要独立 code review 时）

框架不设固定的 Reviewer 角色——每个任务的验收已由独立上下文的 Verifier 核查。若你还需要更主观的代码审查（可维护性、架构一致性、安全隐患），把审查本身写成一个普通任务即可：派出去的 Worker 天然就是一个不带实现者偏见的干净上下文 reviewer。

```yaml
id: T9
depends_on: [T1]          # 审查 T1 的产出
touches:
  - docs/reviews/T1-review.md   # 只允许写审查报告, 被审代码全部只读
acceptance:
  - docs/reviews/T1-review.md 存在, 包含「问题清单」与「改进建议」两节
  - 问题清单中每条注明 文件:行号 与严重程度 (高/中/低)
```

审查发现的问题要转化为行动时，写进后续修复任务的正文（或由用户追加新任务），而不是让 reviewer 直接改代码。
