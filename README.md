# Loop Engineering — 自主循环工作流框架

一个**零代码、纯提示词 + 文件协议**的自主智能体循环框架。给 LLM 一份标准化任务清单，它自动接管：拆解调度 → 派发执行 → 验收反馈 → 更新状态，循环直至完成。

**快速开始**：见 [LOOP.md](LOOP.md)。本文档解释设计。

---

## 一、设计理念

### 1. 编排智能在提示词，状态可靠性在文件系统

传统 agent 框架把编排逻辑写进代码（LangChain、AutoGPT 式的编排器）。本框架反其道而行：**调度决策交给 LLM 本身**（它比 if-else 更懂"这个任务失败的原因是否值得换个思路重试"），而**可靠性交给文件系统**（state.md 不会失忆，回执不会被上下文压缩吞掉）。于是移植成本降到极限——复制几个 markdown 文件即可，无依赖、无 API key、无运行时。

### 2. 上下文是稀缺资源，按角色隔离预算

长循环工作的头号杀手是上下文窗口耗尽与信息退化。本框架的对策是把上下文当作显式管理的资源：

| 角色 | 上下文预算 | 纪律 |
|---|---|---|
| 主控 | 全循环存活，最稀缺 | 只读清单/状态/回执，禁止执行任务、禁止读产出物全文 |
| Worker | 单任务存活，用完即弃 | 只读自己的简报，任务再复杂也不污染主控 |
| 文件系统 | 无限、持久 | 一切跨轮记忆的唯一载体 |

信息在角色间流动时**强制过窄口径**：主控→Worker 只有简报，Worker→主控只有 ≤60 行的结构化回执。这个"信息漏斗"是刻意的——它逼迫每一层只传递决策所需的最小信息，从结构上杜绝上下文膨胀。

### 3. 假设随时失忆，设计就自然健壮

框架不试图"避免中断"，而是让中断变得无所谓：状态先写后做、每轮落盘、冷启动协议对账续跑。上下文耗尽和人为中断走同一条恢复路径。这借鉴了分布式系统的 crash-only design：与其处理优雅关闭，不如让崩溃恢复成为唯一路径。

### 4. 与业界方案的关系

- **Plan-and-Solve**：全局规划前置——但规划由人类在编写 TASKS.md 时完成（依赖图、验收标准），LLM 不在运行时重新发明计划。经验表明运行时自主规划是 AutoGPT 类系统跑偏的主因。
- **ReAct**：Worker 内部采用思考-行动交替（见 `prompts/worker.md` 工作流程）。
- **Reflexion**：验收失败后，主控写出失败反思注入重试简报，而非原样重发。
- **Orchestrator-Worker**（Anthropic 多智能体研究系统的核心模式）：主从分离 + 结构化交接。

## 二、整体架构

```
┌─────────────────────────────────────────────────┐
│  TASKS.md  任务清单（人类编写, 唯一输入, 只读）      │
└──────────────────────┬──────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────┐
│  主控 Orchestrator（薄调度层）                     │
│  读状态 → 选就绪任务(依赖检查) → 写简报 → 派发      │
│  → 验收回执 → 更新 state.md → 下一轮              │
└───────┬──────────────────────────────▲──────────┘
        │ briefs/<id>.md               │ reports/<id>.md
        │ (简报: 目标/上游交接/          │ (回执: 自查/产出/
        │  验收标准/写入范围)            │  关键决策/下游提醒)
        ▼                              │
┌─────────────────────────────────────────────────┐
│  Worker × N（子代理, 一次性上下文, 可并行）         │
└─────────────────────────────────────────────────┘
        ▲                              │
        └────────── loop/state.md ─────┘
           唯一事实源: 状态表 / 决策日志 / 备忘
           （先写后做 + 每轮落盘 → 任意中断可恢复）
```

## 三、三大核心问题的解法

### 上下文管理

- **状态全外置**：`loop/state.md` 是唯一事实源，新会话读它即可完整接管（`protocol/state-spec.md`）。
- **信息漏斗**：主控不读产出物全文、Worker 不读清单全貌；跨任务信息传递只走"上游回执的关键决策/下游提醒 → 自动注入下游简报"这一条通道。
- **主动断点**：主控自感上下文退化时落盘停机，用户一句「按 LOOP.md 继续」无损续跑。

### 任务分配与调度

- **依赖驱动**：清单中声明 `depends_on`，主控每轮扫描"依赖全部 done 且自身未完成"的就绪任务。启动时做环检测。
- **并行安全**：清单中声明 `touches`（允许写入的路径），touches 无重叠且无依赖关系的就绪任务同批并行派发（Claude Code 多子代理；无子代理环境自动降级为串行角色切换）。
- **粒度前置**：任务拆解在编写清单时由人完成（一个任务 = 一个 Worker 一次上下文的工作量），运行时不做自主再拆解，换取可预测性。

### 主从协同

- **纯文件协议**：简报（主控→Worker）与回执（Worker→主控）是唯二通道，格式见 `protocol/handoff-spec.md`。
- **验收不盲信**：Worker 逐条自查 + 主控对可机械验证的标准抽查（跑命令/查文件/grep）。
- **失败闭环**：验收失败 → 主控写反思 → 注入重试简报（≤2 次重试）→ 仍败标记 blocked → 不影响其他任务推进 → 最终汇总升级人工。凭据缺失、清单矛盾、破坏性操作三类问题立即升级。

## 五、适用边界（诚实版）

- **适合**：可分解为 5–30 个中等粒度任务、验收标准可显式写出的项目——代码开发、文档工程、内容生产、数据整理。
- **不适合**：验收标准无法预先定义的探索性工作；任务间深度纠缠无法划清 `touches` 的重构；需要毫秒级协同的实时系统。
- **信任模型**：框架靠提示词纪律约束 LLM，不是硬沙箱。高风险操作（删除、部署、花钱）应在清单 `constraints` 中显式禁止，并依赖宿主工具（如 Claude Code 权限系统）兜底。
