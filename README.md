# Loop Engineering — 自主循环工作流框架

一个**零代码、纯提示词 + 文件协议**的自主智能体循环框架。给 LLM 一份标准化任务清单，它自动接管：拆解调度 → 隔离执行 → 独立验收 → 合并落地 → 回归保护，循环直至完成。

**快速开始**：见 [LOOP.md](LOOP.md)。已安装配套 skill 的 Claude Code 用户更简单：`/loop-engineering <你的需求>` 一句话即可（skill 会自动部署框架、生成任务清单并运行循环；skill 定义见 [skill/SKILL.md](skill/SKILL.md)，安装 = 将其与框架文件放入 `~/.claude/skills/loop-engineering/`）。本文档解释设计。

---

## 一、设计理念

### 1. 编排智能在提示词，可靠性在事件日志

传统 agent 框架把编排逻辑写进代码（LangChain、AutoGPT 式的编排器）。本框架反其道而行：**调度判断交给 LLM**（它比 if-else 更懂"这个任务失败的原因是否值得换个思路重试"），而**可靠性交给一个只追加的事件日志**（`loop/events.jsonl`，借鉴数据库 WAL 与事件溯源）。任务状态不存储在任何地方——它永远由重放事件流计算得出：追加一行是原子的，重放是机械的，历史统计是免费的。移植成本降到极限——复制几个 markdown 文件即可，无依赖、无 API key、无运行时。

### 2. 一切智能组件都是一次性的，只有文件永生

长循环工作的头号杀手是上下文窗口耗尽与信息退化。本框架的答案是**取消长驻上下文这个概念本身**——主控不是守一整场的长驻会话，而是以 **tick** 为单位工作的一次性上下文（一个 tick = 重放事件流 → 收租约 → 验收落地 → 派发 → 落盘事件）：

| 角色 | 生命周期 | 纪律 |
|---|---|---|
| 主控 tick | 单次调度决策 | 只读事件流/回执/裁决，禁止执行任务、读产出物全文、亲自验证 |
| Worker | 单任务 | 只读自己的简报，在隔离工作区执行，任务再复杂也不污染他人 |
| Verifier | 单次验证 | 独立核查，消化构建日志与产出物全文，只交回 ≤30 行裁决 |
| 事件日志 | 永生 | 唯一事实源，一切跨 tick 记忆的唯一载体 |

信息在角色间流动时**强制过窄口径**：主控→Worker 只有简报，Worker→主控只有 ≤60 行回执，Verifier→主控只有 ≤30 行裁决。这个"信息漏斗"是刻意的——它逼迫每一层只传递决策所需的最小信息，从结构上杜绝上下文膨胀。主控自身的上下文退化问题则**在结构上消失了**：它随时可以结束当前 tick，任何新上下文从事件流无损接管。

### 3. 崩溃恢复即正常运行（crash-only design）

框架不区分"正常运行"与"故障恢复"——每个 tick 都从重放事件流开始，冷启动只是碰巧上一 tick 在另一个会话里。崩溃、人为中断、Worker 超时挂死（租约机制检测），三种故障走**同一条恢复路径**：丢弃隔离工作区、按瞬时失败免费重派。不为不同故障发明不同恢复分支。

### 4. 每次尝试是一个事务（Saga 式回滚）

每次任务尝试在隔离的 git worktree 中执行：验收通过才合并主干，不通过整个丢弃——**主干永远干净，回滚零成本，失败零污染**。并行安全由此从"事前悲观锁"（指望人类把 `touches` 写对）变成"事后乐观检测"（写错了在合并时显式暴露为冲突，丢弃后到者重做，而非静默数据损坏）。文件系统之外的副作用（API 调用、部署）事务管不了——用 `effects` 声明 + 副作用日志 + 重试注入实现幂等。

### 5. 验收即代码，done 是持续成立的性质

验收标准中可机械化的部分被 Verifier 编译成每任务一个的 checks 脚本（`loop/checks/<id>.sh`）：重试复验免费、每次落地后重跑相关脚本提供**回归保护**（T5 改坏 T1 已验收的功能会被当场抓住）、循环结束后整个目录作为验收测试套件交付。LLM 判断只花在无法脚本化的语义核查上。

### 6. 人在环上，而非环中

需要人类决策时（缺凭据、清单矛盾、任务需拆分），主控写一个问题文件（`loop/questions/`）后**继续推进其他任务**；人类随时作答，下一 tick 自动恢复。人类的指令走 `loop/inbox/`，修订 `TASKS.md` 每 tick 自动同步——升级不停机，干预不等待。

### 7. 与业界方案的关系

- **事件溯源 / WAL**（数据库、LangGraph checkpointer）：只追加日志 + 状态重放。
- **无状态 reconcile loop**（Kubernetes controller、Temporal）：每 tick 从事实源重算，不依赖上次循环的内存。
- **隔离工作区 + 合并落地**（Cursor Background Agent、Claude Code worktree）：每尝试一个分支，PR 式落地。
- **Saga / 租约 / 幂等**（分布式系统）：统一补偿动作（丢弃工作区）、超时故障检测、副作用日志防重放。
- **Orchestrator-Worker + 结构化交接**（Anthropic 多智能体研究系统）：主从分离、artifacts 优先于转述（接口契约走产物文件）。
- **Plan-and-Solve**：全局规划由人类在编写 TASKS.md 时前置完成，LLM 不在运行时重新发明计划（运行时自主规划是 AutoGPT 类系统跑偏的主因）。
- **ReAct / Reflexion**：Worker 内部思考-行动交替；验收失败后带反思重试而非原样重发。

## 二、整体架构

```
TASKS.md（人写, 唯一输入, 每tick自动同步修订）   loop/inbox/（人类异步指令）
      │                                            │
      ▼                                            ▼
┌────────────────────────────────────────────────────────┐
│ Orchestrator Tick（一次性上下文, 每 tick 新生）            │
│ 重放 events.jsonl → 消费 inbox/questions → 收租约        │
│ → 验收(读回执/派Verifier/读裁决) → 合并落地+回归            │
│ → 选就绪任务 → 组装简报 → 派发 → 追加事件 → 消亡           │
└───────┬────────────────────────────────────────────────┘
        │ 每任务尝试 = 一个事务（隔离 git worktree）
        ▼
   Worker 执行(ReAct) ──回执──> 验收 ──通过──> merge 主干 + 回归 checks
        │                        │
        │                     不通过 → 丢弃 worktree（主干零污染）
        ▼                        ▼
┌─────────────────────────────────────────────────────────┐
│ loop/events.jsonl   唯一事实源（append-only, 原子, 可重放） │
│   ├── state.md / decisions.md   ← 投影（人类视图, 可重算）  │
│   ├── checks/    ← 机械验收脚本（回归保护, 最终交付）        │
│   └── questions/ ← 异步人机通道（升级不停机）               │
└─────────────────────────────────────────────────────────┘
```

## 三、三大核心问题的解法

### 上下文管理

- **状态全外置且免维护**：事实源是事件日志，状态靠重放；`state.md` 只是投影，损坏无害。
- **主控 tick 化**：不存在"主控上下文耗尽"问题——tick 之间天然是断点，续跑免费。
- **信息漏斗**：主控不读产出物全文、Worker 不读清单全貌；跨任务信息走三条受控通道——上游回执摘要注入下游简报（主通道）、机器可读契约文件（接口约定优先走产物而非转述）、带 scope 标签的决策索引（按相关性筛选注入，而非全文灌入）。

### 任务分配与调度

- **依赖驱动**：清单声明 `depends_on`，每 tick 扫描就绪任务，关键路径（后继依赖链最长）优先。启动时做环检测。
- **并行安全双保险**：事前 `touches`/`resources` 声明避免大多数冲突；事后 worktree 合并检出真实冲突（丢弃后到者重试，不静默损坏）。
- **粒度前置**：任务拆解由人类在编写清单时完成；运行时 Worker 报告任务过大 → 转问题文件请人类拆分，不自主再拆解。

### 主从协同

- **纯文件协议**：简报/回执/裁决三条通道，按 `<id>.attempt<n>.md` 命名，失败历史天然完整保留。
- **验收不盲信**：Worker 逐条自查（诚实自查有激励：自报 ❌ 的首次失败不耗重试）+ 独立 Verifier 核查（checks 脚本 + 语义抽查）；高风险任务（`risk: high`）双 Verifier 分视角核查。
- **失败闭环按类分流**：瞬时失败（超时/冲突/限流）直接重试不受罚；确定性失败带反思重试（≤2 次，反思以裁决的 ❌ 条目与修复建议为依据），仍败 blocked；环境性失败（缺凭据/清单矛盾/需拆分）立即转问题文件等人类，**循环不停**。落地后回归 checks 保证 done 任务不被后续工作静默破坏。

## 五、适用边界（诚实版）

- **适合**：可分解为 5–30 个中等粒度任务、验收标准可显式写出的项目——代码开发、文档工程、内容生产、数据整理。
- **不适合**：验收标准无法预先定义的探索性工作；任务间深度纠缠无法划清 `touches` 的重构；需要毫秒级协同的实时系统。
- **git 依赖**：事务工作区、失败回滚、冲突检测都建立在 git worktree 上。非 git 项目可降级运行（串行 + 无回滚），但强烈建议 `git init`。
- **信任模型**：框架靠提示词纪律约束 LLM，不是硬沙箱。worktree 隔离的是文件系统，隔离不了外部副作用（API 调用、部署）——高风险操作应在清单 `constraints` 中显式禁止、用 `effects` 声明可追踪，并依赖宿主工具（如 Claude Code 权限系统）兜底。
- **协议即规格**：全部机械操作（事件重放、简报组装、worktree 管理、租约检查）均有确定性定义——若你愿意写几百行脚本把它们下沉为 CLI（`loopctl`），协议文档就是现成的规格书；纯提示词运行是同一协议的零依赖降级形态。
