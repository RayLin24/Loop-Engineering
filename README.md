# Loop Engineering — 自主循环工作流框架

一个**纯提示词 + 文件协议**的自主智能体循环框架。给 LLM 一份标准化任务清单，它自动接管：拆解调度 → 派发执行 → 验收反馈 → 更新状态，循环直至完成。主控采用**代际制**——会话是一次性躯壳，记忆全在文件系统，可选驱动器无人值守跑完全程。

**快速开始**：见 [LOOP.md](LOOP.md)。本文档解释设计。

---

## 一、设计理念

### 1. 编排智能在提示词，状态可靠性在文件系统

传统 agent 框架把编排逻辑写进代码（LangChain、AutoGPT 式的编排器）。本框架反其道而行：**调度决策交给 LLM 本身**（它比 if-else 更懂"这个任务失败的原因是否值得换个思路重试"），而**可靠性交给文件系统**（state.md 不会失忆，回执不会被上下文压缩吞掉）。于是移植成本降到极限——复制几个 markdown 文件即可，无依赖、无 API key、无运行时。唯一的可选代码是 `runner/` 驱动脚本：十几行胶水，只做"拉起下一代会话 + 判停"，不含任何调度逻辑。

### 2. 主控的持久性在文件系统，不在会话——代际制

长循环的头号杀手是主控上下文耗尽。早期版本靠主控"自感上下文将尽再断点"，实测不可靠：模型对自己剩余上下文的感知很差，要么反复纠结"要不要停"（纠结本身烧预算），要么硬撑到轮中失忆。现行设计改为**确定性代际轮换**：

- 每代主控会话最多服务「每代最大轮次」轮（默认 3，白名单可调），**到轮就交，不做自我评估**——把上下文管理从元认知问题降为计数问题。
- 交接即三行消息 + 落盘退出；下一代会话经冷启动协议（代际登记 + dispatched 对账）无损接管。
- `state.md` 的 `循环状态` 字段（running/handoff/awaiting-human/finished）是生命周期信号：人类一眼判断进度，`runner/` 驱动器据此决定"继续拉起下一代"还是"停下等人"。
- 驱动器自带两道保险丝：代数上限 + 连续停滞检测（一代跑完轮次没前进，说明在空转，连续两次即停）。

于是任务数不再受单会话上下文约束：每代的消耗有上界，代数没有上界。

### 3. 上下文是稀缺资源，按角色隔离预算

| 角色 | 上下文预算 | 纪律 |
|---|---|---|
| 主控 | 一代之内存活，最稀缺 | 只读清单/状态/回执/裁决/元报告，禁止执行任务、禁止读产出物全文、禁止亲自跑验证 |
| Worker | 单任务存活，用完即弃 | 按简报阅读清单自行装配认知，任务再复杂也不污染主控 |
| Verifier | 单次验证存活，用完即弃 | 独立核查验收标准，消化构建日志与产出物全文，只交回 ≤30 行裁决 |
| Meta-Analyst | 单次分析存活，用完即弃 | 只读轨迹（state.md + 归档回执/裁决），不碰产出物与命令，只交回 ≤40 行元报告 |
| 文件系统 | 无限、持久 | 一切跨轮记忆的唯一载体 |

信息在角色间流动时**强制过窄口径**：主控→Worker 只有简报，Worker→主控只有 ≤60 行的结构化回执（主控只读前 60 行，预算由读者端强制执行），Verifier→主控只有 ≤30 行的裁决。这个"信息漏斗"是刻意的——它逼迫每一层只传递决策所需的最小信息，从结构上杜绝上下文膨胀。

**简报是指针，不是拷贝**。主控写的简报只含「阅读清单」（读 TASKS.md 哪几节、哪些上游回执、decisions/lessons），装配工作由收件的 Worker/Verifier 自己完成。理由是预算不对称：同样的信息装配，发生在一次性上下文里近乎免费，发生在主控身上直接缩短这一代的寿命。主控亲笔的只有判断性内容：附言、命令授权、失败反思。实测这把主控的每任务调度成本压缩到原来的约五分之一。

### 4. 假设随时失忆，设计就自然健壮

框架不试图"避免中断"，而是让中断变得无所谓：状态先写后做、每轮增量落盘、冷启动协议对账续跑。代际交接、上下文耗尽和人为中断走同一条恢复路径。这借鉴了分布式系统的 crash-only design：与其处理优雅关闭，不如让崩溃恢复成为唯一路径——代际轮换正是"主动定期崩溃"。

### 5. 与业界方案的关系

- **Plan-and-Solve**：全局规划前置——但规划由人类在编写 TASKS.md 时完成（依赖图、验收标准），LLM 不在运行时重新发明计划。经验表明运行时自主规划是 AutoGPT 类系统跑偏的主因。同理，主控运行时**不得自行追加任务**——衍生需求只能登记备忘留给人类，防失控自我扩张。
- **ReAct**：Worker 内部采用思考-行动交替（见 `prompts/worker.md` 工作流程）。
- **Reflexion**：验收失败后，主控写出失败反思注入重试简报，而非原样重发。
- **Orchestrator-Worker**（Anthropic 多智能体研究系统的核心模式）：主从分离 + 结构化交接。
- **Bilevel / Autoresearch**（[karpathy/autoresearch](https://github.com/karpathy/autoresearch)、[Bilevel-Autoresearch](https://github.com/EdwardOptimization/Bilevel-Autoresearch)）：内层循环优化任务产出，元循环优化循环自身的运行方式（见 `protocol/meta-spec.md`）。从前者借来「评测不可被优化对象篡改」（prompts/protocol 运行期只读, 对应其 `prepare.py` 只读）与量化台账（state.md 循环度量, 对应其 `results.tsv`）；从后者借来 Level 1.5 / Level 2 分层（配置调整 vs 机制建议）与关键隔离约束「经验只影响提案, 永不影响评判」（lessons 在 Worker 阅读清单中, 在 Verifier 禁读清单中）。与两者的差异：本框架 Level 2 不自改机制——机制建议只留给人类, 因为这里的"机制"是提示词与协议本身, 让循环自改等于让被测者改考卷。

## 二、整体架构

```
┌──────────────────────────────────────────────────────┐
│ runner/ 驱动器 (可选, 纯胶水)                           │
│   while 未完成: claude -p "按 LOOP.md 继续"  ← 每代新会话 │
│   判停: state.md 循环状态 / FINAL.md / 停滞检测          │
└──────────────────────┬───────────────────────────────┘
                       ▼ 拉起
┌──────────────────────────────────────────────────────┐
│ 主控 Orchestrator (第 N 代会话, ≤每代最大轮次 后交接)      │
│   读 TASKS.md+state.md → 选就绪任务(依赖+touches+       │
│   resources) → 写指针简报 → 派发 → 读回执(≤60行)         │
│   → 派 Verifier → 读裁决(≤30行) → 增量落盘 → 下一轮      │
└───┬──────────────▲───────┬──────────────▲────────────┘
    │ briefs/      │ reports/  briefs/    │ verdicts/
    │ <id>.md      │ <id>.md   <id>.verify.md  <id>.md
    │ (指针简报)    │ (回执)    │ (指针验证简报) │ (裁决)
    ▼              │          ▼           │
┌──────────────────────┐  ┌──────────────────────┐
│ Worker × N           │  │ Verifier             │
│ (一次性上下文, 按阅读   │  │ (一次性上下文, 按验证   │
│  清单自行装配, 可并行)  │  │  依据独立核查+禁读清单) │
└──────────────────────┘  └──────────────────────┘
        ▲                              │
        └────────── loop/state.md ─────┘
           唯一事实源: 循环状态 / 状态表 / 日志 / 备忘
           （先写后做 + 每轮增量落盘 → 任意中断可恢复）
```

## 三、三大核心问题的解法

### 上下文管理

- **代际制主控**：每代固定轮数强制交接（硬规则，不靠自我感知），`runner/` 自动接续；单代消耗有上界，总任务数无上界。
- **状态全外置**：`loop/state.md` 是唯一事实源，新会话读它即可完整接管（`protocol/state-spec.md`）；落盘一律增量编辑，禁止全文重写。
- **信息漏斗 + 指针简报**：主控不读产出物全文、不抄写简报内容；跨任务信息传递走两条受控通道——直接依赖的"上游回执摘要"与全局性的 `loop/decisions.md`（只追加，超 40 行才压缩一次），均由 Worker 按阅读清单自取。
- **播报纪律**：主控对用户每轮 ≤3 行，长输出只出现在交接/升级/终局三处。

### 任务分配与调度

- **依赖驱动**：清单中声明 `depends_on`，主控每轮扫描"依赖全部 done 且自身未完成"的就绪任务。启动时做环检测。
- **并行安全**：清单中声明 `touches`（允许写入的路径）与可选的 `resources`（独占共享资源, 如端口、lockfile、**构建产物目录**），两者均无重叠且无依赖关系的就绪任务同批并行派发。资源互斥**覆盖验证阶段**：占用从派发 Worker 持续到读回裁决——两个 Verifier 并行跑同一构建会因产物写入交错产生假失败，这是实测教训。
- **粒度前置**：任务拆解在编写清单时由人完成（一个任务 = 一个 Worker 一次上下文的工作量），运行时不做自主再拆解，换取可预测性。

### 主从协同

- **纯文件协议**：简报（主控→Worker）、回执（Worker→主控）、裁决（Verifier→主控）是仅有的三条通道，格式见 `protocol/handoff-spec.md`。重试前旧文件归档为 `<id>.attempt<n>.md`，失败历史完整可查。
- **验收不盲信**：Worker 逐条自查 + 独立上下文的 Verifier 逐条核查（跑命令、读产出物、查约束与 touches 越界），主控只依据 ≤30 行的裁决定 done/failed。诚实自查有激励：Worker 自报 ❌ 的首次失败不消耗重试次数。
- **裁决也不盲信**：裁决与自查矛盾且证据像环境噪音、或同轮裁决对同一事实冲突时，主控走「可疑裁决复核」——重派全新 Verifier（简报注明先 clean）仲裁，不消耗重试次数。主控在任何情况下不亲自读产出物仲裁：实测一次亲自仲裁的预算 ≈ 三个任务的正常调度。
- **失败闭环**：验收失败 → 主控写反思（以裁决 ❌ 条目与 Verifier 的「修复建议」为依据——Verifier 是唯一读过产出物全文的角色）→ 注入重试简报（≤2 次重试）→ 仍败标记 blocked → 不影响其他任务推进 → 最终汇总升级人工。凭据缺失、清单矛盾、破坏性操作、任务超出单次上下文四类问题立即升级。
- **元循环**（`protocol/meta-spec.md`）：每 5 轮、任务 blocked 或同类失败 ≥3 次时，主控派发一次性 Meta-Analyst 分析循环轨迹——在白名单内调整运行配置（并行度、重试上限、每代最大轮次、lessons 开关、验证强度），全量重写 `loop/lessons.md`（跨任务失败模式, 进后续 Worker 阅读清单），并把对 prompts/protocol/清单写法的结构性建议写进元报告留给人类。三条硬边界：机制文件运行期只读、lessons 永不进 Verifier 视野、配置调整不越白名单。

## 五、适用边界（诚实版）

- **适合**：可分解为中等粒度任务、验收标准可显式写出的项目——代码开发、文档工程、内容生产、数据整理。代际制下任务数不再受单会话限制，几十个任务的清单可无人值守跑完。
- **不适合**：验收标准无法预先定义的探索性工作；任务间深度纠缠无法划清 `touches` 的重构；需要毫秒级协同的实时系统。
- **信任模型**：框架靠提示词纪律约束 LLM，不是硬沙箱。高风险操作（删除、部署、花钱）应在清单 `constraints` 中显式禁止，并依赖宿主工具（如 Claude Code 权限系统）兜底；无人值守模式的权限取舍见 `runner/README.md`。
