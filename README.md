# Loop Engineering — 自主循环工作流框架

一个**纯提示词 + 文件协议**的自主智能体循环框架。给 LLM 一份标准化任务清单，它自动接管：拆解调度 → 派发执行 → 验收反馈 → 合并落地 → 回归保护，循环直至完成。全部状态是一条只追加的事件流——会话是一次性躯壳，记忆全在文件系统，可选驱动器无人值守跑完全程。

**快速开始**：见 [LOOP.md](LOOP.md)。本文档解释设计。

---

## 一、设计理念

### 1. 编排智能在提示词，状态可靠性在文件系统

传统 agent 框架把编排逻辑写进代码（LangChain、AutoGPT 式的编排器）。本框架反其道而行：**调度决策交给 LLM 本身**（它比 if-else 更懂"这个任务失败的原因是否值得换个思路重试"），而**可靠性交给文件系统**（事件流不会失忆，回执不会被上下文压缩吞掉）。于是移植成本降到极限——复制几个 markdown 文件即可，无框架依赖、无 API key、无运行时。唯一的硬依赖是 **git**（工作区事务，见下）与 `runner/` 里十几行胶水脚本：只做"拉起下一个会话 + 判停"，不含任何调度逻辑。

### 2. 状态的唯一事实源是事件流——崩溃恢复是计算，不是纪律

`loop/events.jsonl` 是一条只追加的事件日志：派发、回执、裁决、落地、失败、配置变更，一切事实都是一行 JSON。**任何状态都不存储，永远由重放事件流算出**。这带来三个性质：

- **写入原子**。追加一行最坏丢半行（重放时容忍末行残缺），而全量重写状态表时中断 = 状态损坏。
- **恢复机械**。会话交接、上下文耗尽、人为中断、超时挂死走同一条恢复路径：重放。不依赖"LLM 按协议对账"的纪律——这是 crash-only design 的直接应用：与其处理优雅关闭，不如让崩溃恢复成为唯一路径。
- **历史免费**。尝试次数、失败分类、wall-clock 耗时全部可从事件流统计，元循环的输入因此完备。

### 3. 主控的持久性在事件流，不在会话——确定性轮换

长循环的头号杀手是主控上下文耗尽。框架把上下文管理从元认知问题降为计数问题：每个主控会话最多服务「每会话 tick 上限」轮（默认 3，白名单可调），**到轮就交，不做自我评估**——落盘退出，下一个会话重放事件流无损接管。投影文件 `loop/state.md` 的 `状态` 字段（running/waiting-human/finished）是生命周期信号：人类一眼判断进度，`runner/` 驱动器据此决定"继续拉起"还是"停下等人"。驱动器自带两道保险丝：代数上限 + 连续停滞检测。

于是任务数不再受单会话上下文约束：每个会话的消耗有上界，会话数没有上界。

### 4. 上下文是稀缺资源，按角色隔离预算

| 角色 | 上下文预算 | 纪律 |
|---|---|---|
| 主控 | 一个会话内存活，最稀缺 | 只读清单/事件流/回执/裁决/元报告，禁止执行任务、禁止读产出物全文、禁止亲自验证 |
| Worker | 单任务存活，用完即弃 | 按简报阅读清单自行装配认知，任务再复杂也不污染主控 |
| Verifier | 单次验证存活，用完即弃 | 独立核查验收标准，消化构建日志与产出物全文，只交回 ≤30 行裁决 |
| Meta-Analyst | 单次分析存活，用完即弃 | 只读轨迹（投影 + 事件流 + 归档回执/裁决），不碰产出物与命令，只交回 ≤40 行元报告 |
| 文件系统 | 无限、持久 | 一切跨轮记忆的唯一载体 |

信息在角色间流动时**强制过窄口径**：主控→Worker 只有简报，Worker→主控只有 ≤60 行的结构化回执（主控只读前 60 行，预算由读者端强制执行），Verifier→主控只有 ≤30 行的裁决。这个"信息漏斗"是刻意的——它逼迫每一层只传递决策所需的最小信息，从结构上杜绝上下文膨胀。

**简报是指针，不是拷贝**。主控写的简报只含「阅读清单」（读 TASKS.md 哪几节、哪些上游回执、decisions 的哪些 scope、lessons），装配工作由收件的 Worker/Verifier 自己完成。理由是预算不对称：同样的信息装配，发生在一次性上下文里近乎免费，发生在主控身上直接缩短这个会话的寿命。主控亲笔的只有判断性内容：附言、命令授权、失败反思。跨任务决策以 `[scope]` 标签索引，Worker 只读取与自己领域匹配的条目——不全文注入。

### 5. 假设随时失忆，设计就自然健壮

框架不试图"避免中断"，而是让中断变得无所谓：先写事件后派发、每轮重算投影、租约超时即故障检测。每个任务的每次尝试在**独立 git worktree** 中执行：越界写入在合并前被 `git diff` 结构化拦截，并行冲突显式化为合并冲突（先到先落地，后到者基于新主干重来），失败尝试整片丢弃——工作区事务把 v1 靠提示词纪律维持的不变量变成了 git 保证的不变量。

### 6. 与业界方案的关系

- **Plan-and-Solve**：全局规划前置——但规划由人类在编写 TASKS.md 时完成（依赖图、验收标准），LLM 不在运行时重新发明计划。经验表明运行时自主规划是 AutoGPT 类系统跑偏的主因。同理，主控运行时**不得自行追加任务**——衍生需求只能登记 note 事件留给人类，防失控自我扩张。
- **ReAct**：Worker 内部采用思考-行动交替（见 `prompts/worker.md` 工作流程）。
- **Reflexion**：验收失败后，主控写出失败反思注入重试简报，而非原样重发。
- **Orchestrator-Worker**（Anthropic 多智能体研究系统的核心模式）：主从分离 + 结构化交接。
- **Bilevel / Autoresearch**（[karpathy/autoresearch](https://github.com/karpathy/autoresearch)、[Bilevel-Autoresearch](https://github.com/EdwardOptimization/Bilevel-Autoresearch)）：内层循环优化任务产出，元循环优化循环自身的运行方式（见 `protocol/meta-spec.md`）。从前者借来「评测不可被优化对象篡改」（prompts/protocol 运行期只读, 对应其 `prepare.py` 只读）与量化台账（事件流 + 循环度量, 对应其 `results.tsv`）；从后者借来 Level 1.5 / Level 2 分层（配置调整 vs 机制建议）与关键隔离约束「经验只影响提案, 永不影响评判」（lessons 在 Worker 阅读清单中, 在 Verifier 禁读清单中）。与两者的差异：本框架 Level 2 不自改机制——机制建议只留给人类, 因为这里的"机制"是提示词与协议本身, 让循环自改等于让被测者改考卷。

## 二、整体架构

```
┌──────────────────────────────────────────────────────────┐
│ runner/ 驱动器 (可选, 纯胶水)                              │
│   while 未完成: claude -p "按 LOOP.md 继续"  ← 每会话新会话  │
│   判停: state.md 投影 状态字段 / FINAL.md / 停滞检测         │
└──────────────────────┬───────────────────────────────────┘
                       ▼ 拉起
┌──────────────────────────────────────────────────────────┐
│ 主控 Orchestrator (第 N 个会话, ≤每会话 tick 上限 后交接)     │
│   重放 events.jsonl → 选就绪任务(依赖+touches+resources)     │
│   → 建 worktree → 写指针简报 → 派发 → 读回执(≤60行)          │
│   → 派 Verifier → 读裁决(≤30行) → 越界检查 → merge 落地      │
│   → 回归 checks → 追加事件 → 重算投影 → 下一轮               │
└───┬──────────────▲───────┬──────────────▲────────────────┘
    │ briefs/      │ reports/  briefs/    │ verdicts/
    │ <id>.md      │ <id>.md   <id>.verify.md  <id>.md
    │ (指针简报)    │ (回执)    │ (指针验证简报) │ (裁决)
    ▼              │          ▼           │
┌──────────────────────┐  ┌──────────────────────┐
│ Worker × N           │  │ Verifier             │
│ (一次性上下文, 独立    │  │ (一次性上下文, 在任务   │
│  worktree 中执行)     │  │  worktree 内独立核查) │
└──────────────────────┘  └──────────────────────┘
        ▲                              │
        └────── loop/events.jsonl ─────┘
           唯一事实源（只追加; 先写后做 → 任意中断可恢复）
           投影: state.md(进度视图) / decisions.md(决策索引)
           人机通道: questions/(主控提问) + inbox/(人类指令)
```

## 三、三大核心问题的解法

### 上下文管理

- **会话轮换**：每个会话固定轮数强制交接（硬规则，不靠自我感知），`runner/` 自动接续；单会话消耗有上界，总任务数无上界。
- **状态全外置**：`loop/events.jsonl` 是唯一事实源，新会话重放即可完整接管（`protocol/event-spec.md`）；投影每轮重算，损坏无害。
- **信息漏斗 + 指针简报**：主控不读产出物全文、不抄写简报内容；跨任务信息传递走两条受控通道——直接依赖的"上游回执摘要"与按 scope 筛选的 `loop/decisions.md` 投影，均由 Worker 按阅读清单自取。
- **播报纪律**：主控对用户每轮 ≤3 行，长输出只出现在交接/升级/终局三处。

### 任务分配与调度

- **依赖驱动**：清单中声明 `depends_on`，主控每轮扫描"依赖全部 done 且自身未完成"的就绪任务。启动时做环检测。
- **并行安全**：清单中声明 `touches`（允许写入的路径）与可选的 `resources`（独占共享资源, 如端口、lockfile、数据库），两者均无重叠且无依赖关系的就绪任务同批并行派发。每个任务在独立 git worktree 中执行——构建产物天然隔离，资源互斥只管端口/数据库这类真共享资源，且**覆盖验证阶段**（占用从派发 Worker 持续到读回裁决）。
- **粒度前置**：任务拆解在编写清单时由人完成（一个任务 = 一个 Worker 一次上下文的工作量），运行时不做自主再拆解，换取可预测性。

### 主从协同

- **纯文件协议**：简报（主控→Worker）、回执（Worker→主控）、裁决（Verifier→主控）是仅有的三条任务通道，格式见 `protocol/handoff-spec.md`；另有 `questions/`（主控→人类提问）与 `inbox/`（人类→主控指令）两条人机通道。重试前旧文件归档为 `<id>.attempt<n>.md`，失败历史完整可查。
- **验收不盲信**：Worker 逐条自查 + 独立上下文的 Verifier 逐条核查（跑命令、读产出物、查约束），主控只依据 ≤30 行的裁决定 done/failed。诚实自查有激励：Worker 自报 ❌ 的首次失败不消耗重试次数。
- **落地不失控**：裁决通过后还要过两道闸——合并前 `git diff` 越界检查（产出物必须是 touches 子集）、合并后全量 checks 回归（已验收任务的机械标准不被后续落地破坏，破坏者负责修复）。裁决也不盲信：证据像环境噪音时走「可疑裁决复核」重派 Verifier 仲裁，不消耗重试次数。
- **失败闭环**：验收失败 → 主控写反思（以裁决 ❌ 条目与 Verifier 的「修复建议」为依据）→ 注入重试简报（默认 ≤2 次）→ 仍败标记 blocked → 不影响其他任务推进 → 汇总升级人工（inbox 指令解救）。凭据缺失、清单矛盾、破坏性操作、任务超出单次上下文四类问题走 questions 通道挂起等待人类回答——提问只挂起单个任务，不停摆整个循环。
- **元循环**（`protocol/meta-spec.md`）：每 5 轮、任务 blocked 或同类失败 ≥3 次时，主控派发一次性 Meta-Analyst 分析循环轨迹（投影 + 事件流 + 归档）——在白名单内调整运行配置（并行度、重试上限、每会话 tick 上限、lessons 开关、验证强度），全量重写 `loop/lessons.md`（跨任务失败模式, 进后续 Worker 阅读清单），并把对 prompts/protocol/清单写法的结构性建议写进元报告留给人类。三条硬边界：机制文件运行期只读、lessons 永不进 Verifier 视野、配置调整不越白名单。

## 五、适用边界（诚实版）

- **适合**：可分解为中等粒度任务、验收标准可显式写出的项目——代码开发、文档工程、内容生产、数据整理。会话轮换下任务数不再受单会话限制，几十个任务的清单可无人值守跑完。
- **不适合**：验收标准无法预先定义的探索性工作；任务间深度纠缠无法划清 `touches` 的重构；需要毫秒级协同的实时系统。
- **硬依赖**：git（worktree 事务与落地回归都建立在它之上）；能起子代理的 LLM 宿主（无子代理能力时可降级为单体会话角色切换，代价是上下文隔离失效，见 `prompts/orchestrator.md`）。
- **信任模型**：框架靠提示词纪律 + git 工作区隔离约束 LLM，不是硬沙箱。高风险操作（删除、部署、花钱）应在清单 `constraints`/`effects` 中显式声明，并依赖宿主工具（如 Claude Code 权限系统）兜底；无人值守模式的权限取舍见 `runner/README.md`。
